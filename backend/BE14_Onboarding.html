1. Was ist verifAi / BE-14?

verifAi prüft, ob wissenschaftliche Behauptungen in einem Dokument wirklich durch die zitierten Quellen gestützt werden. Es erkennt erfundene Zitationen, falsch zugeordnete Quellen und Quellen, die die Aussage gar nicht belegen.

BE-14 ist die aktuelle Integrationsstufe. Sie baut auf dem BE-13-Baseline auf und fügt die vollständige Volltext-Pipeline (Unpaywall, arXiv, User-PDF-Upload), Titel-basiertes DOI-Lookup, SSRN-Erkennung und die RAG-Hybrid-Pipeline hinzu.

Für Teammitglieder: Dieser Guide zeigt euch, wie ihr das Projekt lokal zum Laufen bringt — inklusive aller API-Keys, die ihr dafür braucht.

3. API-Keys — Wo bekommt ihr sie, wo kommen sie hin?

Wichtig: Die .env-Datei niemals committen oder teilen. Sie ist in .gitignore eingetragen und bleibt lokal.

3.1 OpenRouter (Pflicht — LLM + Embeddings)

Wir nutzen OpenRouter als einzigen API-Provider. Er gibt uns Zugang zu allen Modellen (Llama 4 Scout für Verifikation, text-embedding-3-small für Embeddings) über einen einzigen Key.

  • Account erstellen unter openrouter.ai
  • Dashboard → "Keys" → "Create Key"
  • Kostenfrei nutzbar mit Guthaben-Limit; für Demo-Zwecke reicht das Free-Tier

In backend/.env eintragen:

OPENROUTER_API_KEY=sk-or-v1-xxxxxxxxxxxxxxxxxxxxxxxx
OPENROUTER_BASE_URL=https://openrouter.ai/api/v1

Außerdem muss in rag/.env (RAG-Subgruppe) derselbe Key stehen: 

OPENROUTER_API_KEY=sk-or-v1-xxxxxxxxxxxxxxxxxxxxxxxx
OPENROUTER_BASE_URL=https://openrouter.ai/api/v1

3.2 Unpaywall (kostenlos, nur E-Mail nötig)

Unpaywall findet legale Open-Access-PDFs zu DOIs. Kein Account, kein Key — nur eine Kontakt-E-Mail laut Unpaywall-Nutzungsbedingungen. 

UNPAYWALL_EMAIL=eure-email@beispiel.de

3.3 CORE API (optional — vollständigere OA-Abdeckung)

CORE erweitert die Volltext-Suche über Unpaywall hinaus. Für die Demo nicht zwingend nötig, erhöht aber die Trefferquote bei Open-Access-Papieren.

  • Kostenloser Key unter core.ac.uk/services/api
  • Rate Limit: 10 req/sec, 1.000 Requests/Tag (Free-Tier)
CORE_API_KEY=euer_core_key_hier

3.4 Alle relevanten .env-Einträge im Überblick

Diese Werte müssen/sollten in backend/.env gesetzt sein:

Variable Pflicht? Standardwert / Hinweis

4. Was BE-14 kann — Feature-Übersicht

4.1 Evidence-Pipeline (Volltext-Beschaffung)

Das System versucht für jede Referenz so viel Text wie möglich zu beschaffen:

FULL_TEXT_AVAILABLE   ← Unpaywall OA-PDF, arXiv-PDF, CORE-Volltext oder User-Upload   ✅ bestes Ergebnis
PREPRINT_AVAILABLE    ← SSRN Working Paper (Abstract)                          ⚠️ Human Review
ABSTRACT_AVAILABLE    ← OpenAlex / SemanticScholar Abstract
METADATA_ONLY         ← Nur Titel / Autoren / Jahr
SOURCE_UNAVAILABLE    ← DOI löst sich nicht auf

4.2 DOI-Lookup-Kette

Für jede Referenz läuft das System folgende Kette durch:

  1. CrossRef — primäre DOI-Auflösung
  2. OpenAlex — Abstract-Fallback
  3. SemanticScholar — Abstract-Fallback + arXiv-ID-Support
  4. Unpaywall — PDF-URL für Open-Access-Papiere
  5. CORE — Volltext-Fallback nach Unpaywall; liefert Text direkt im JSON (kein PDF-Download nötig); aktiv nur wenn CORE_API_KEY gesetzt
  6. SSRN — Abstract-Scraping für Working Papers (DOI 10.2139/ssrn.*)

4.3 Titel-basiertes DOI-Lookup (vier Quellen)

Viele Papiere in Management, Jura und Geisteswissenschaften haben keine DOI im Literaturverzeichnis. Das System sucht dann per Titel bei CrossRef → OpenAlex → SemanticScholar → CORE und prüft jeden Treffer mit drei Heuristiken (Titel-Ähnlichkeit ≥ 0.95, Jahres-Abgleich, Autoren-Überlappung). CORE wird nur einbezogen, wenn CORE_API_KEY in .env gesetzt ist.

4.4 SSRN / Preprint-Erkennung

SSRN-Papiere (DOI-Prefix 10.2139/ssrn.*) werden automatisch erkannt und als PREPRINT_AVAILABLE klassifiziert. Die Safety Policy begrenzt die Konfidenz auf 0.65 und setzt immer human_review_required = true, da der Preprint-Text vom publizierten Paper abweichen kann.

4.5 User-PDF-Upload

Für paywalled Papiere, die Unpaywall nicht erreicht (z.B. Elsevier): der Nutzer kann das PDF manuell hochladen. 

POST /api/v1/references/{reference_id}/upload-source-pdf

4.6 RAG-Pipeline (Retrieval)

Door 1 nutzt Hybrid-Retrieval:

  • Dense FAISS — Cosine-Similarity auf OpenAI-Embeddings
  • BM25 — Keyword-Suche
  • RRF — Reciprocal Rank Fusion zur Zusammenführung
  • FlashRank — Neural Reranking der Top-Kandidaten
  • Embedding Cache — jedes Paper wird nur einmal eingebettet, auch wenn mehrere Claims es zitieren

5. Wichtige API-Endpunkte

Methode Pfad Was es tut
POST /documents/text Dokument hochladen (Text)
POST /documents/{id}/extract-references Referenzen extrahieren
POST /documents/{id}/verify-dois DOIs auflösen + Metadaten holen
POST /documents/{id}/extract-claims Claims aus dem Text extrahieren
POST /documents/{id}/prepare-evidence Evidence Packages bauen
POST /documents/{id}/pipeline-runs Komplette Verifikations-Pipeline starten
GET /documents/{id}/verification-results Ergebnisse abrufen
POST /references/{id}/upload-source-pdf PDF manuell hochladen (Paywall-Fallback)

6. Offline testen (ohne API-Keys)

Für reine Backend-Logik-Tests ohne echte LLM/Embedding-Calls den Mock-Mode aktivieren (bereits Default in .env.example): 

GENAI_MOCK_MODE=true
RAG_MOCK_MODE=true
Im Mock-Mode laufen alle 130 Tests durch — kein OpenRouter-Key nötig.

Für die echte Pipeline (Demo, Live-Verifikation):

GENAI_MOCK_MODE=false
RAG_MOCK_MODE=false

7. Bekannte Einschränkungen

Einschränkung Workaround
Elsevier-PDFs: Unpaywall liefert keine PDF-URL trotz OA-Status PDF manuell über Upload-Endpoint hochladen
Erstes Claim pro Paper dauert 10–25 Sekunden (Embedding + LLM) Verification Cache: Wiederholungen < 5 Sekunden
SSRN-Abstracts können vom publizierten Text abweichen Safety Policy begrenzt Konfidenz auf 0.65, setzt Human-Review-Flag
Semantischer Cache (BE-8) ist noch ein Stub Exakter Cache (claim_id-basiert) funktioniert vollständig

8. Ausblick / TBD

HyDE — Hypothetical Document Embeddings

Statt den Claim direkt als Embedding-Query zu nutzen, generiert ein LLM zuerst einen hypothetischen Absatz, der den Claim stützen würde. Dieser Absatz liegt semantisch näher an den Quell-Chunks (Abstract/Volltext) und verbessert so die Retrieval-Qualität.

Trade-off: +1 LLM-Call pro Claim vor dem Embedding; bei schlechten Hypothesen kann die Qualität sinken. A/B-Test über rag/evaluation/benchmark.py nötig.

Paralleles Embedding

Beim ersten Claim für ein neues Paper laufen Source-Embedding und Query-Embedding aktuell sequenziell. Parallelisierung mit ThreadPoolExecutor würde Door-1-Latenz um ~40–50 % senken — aber das Risiko von Rate-Limit-Fehlern steigt.
